HTML5 기반 UI 전환 프로젝트 內 EDMS 파트 PL

프로젝트 개요

  • 기존 외부 ActiveX 기반으로 제공되던 EDMS 문서관리 기능을 내부 시스템에 내재화하여 유지보수성 및 확장성 개선 프로젝트

기간

  • UI 프로젝트 총 기간

    • 2025.01 ~ 2025.12

  • EDMS 기간

    • 2025.12 ~ 2026.03

담당 역할

  • 프로젝트 일정 및 개발 진척도 관리
  • 외주 개발자와의 커뮤니케이션 및 산출물 검토
  • 개발 이슈 및 리스크 관리
  • 운영 안정화 및 장애 대응 총괄

기술 및 개발 환경

  • Language : Java, SQL
  • Framework : Spring
  • Database : Oracle
  • Middleware : JEUS
  • Solution : BXM, Frism, WebSquare, ImageViewer

시스템 구조

  • 저장

    • 사용자 → UI서버(WebSquare) → APP서버(Service) → Xvarm서버(Save) → DB insert

  • 조회

    • 사용자 → UI서버(WebSquare) → APP서버(Service) → Xvarm서버(Down) → 배치서버(Save) → OCR서버(Viewer)

  • 수정

    • 사용자 → UI서버(WebSquare) → APP서버(Service) → Xvarm서버(Update/Delete)

주요 기능

  • EDMS문서 조회/저장/수정/삭제
  • 권한별 기능 조작

주요 화면

ImageViewer 화면

사용자가 문서를 조회/저장/수정/삭제가 가능한 화면

기술적 이슈 및 해결

문제1

  • 상황

    • Batch reader에서 평일이면서 영업일이 아닌 경우 배치 실패

  • 원인

    • 비영업일이면 retrun값 없이 reader 종료

  • 해결

    • - 평일이면서 영업일이 아닌 경우 배치 실패 처리되지 않도록 return값을 null로 추가하여 reader stepd을 Complite시킴

  • 결과

    • retrun null로 인해 후속 procession 수행되지 않고 정상 종료 처리


문제2

  • 상황

    • 실시간 중앙회 전문통신 시 여신 원장 업데이트 항목에 차기원금납입일이 누락되어 연체시작일이 오조회 됨

  • 원인

    • 환원데이터와 실시간데이터의 D-1 간극

  • 해결

    • - 전문송신 받은 차기원금납입일 항목을 원장에 업데이트하여 납입예정일을 정상 산출함

    - WebToB서버 설정파일 변경 후 재기동 수행

  • 결과

    • 연체시작일 정상 조회

성과

  • ActiveX 기반 프로그램을 내부 소스코드로 수정하여 유지보수 용이(코드 수정 기간 7영업일 → 1영업일)
  • 기능별 권한 확인 부분을 추가하여 고객 정보 관리 보안 강화

회고

배운점

  • 서버 간 API 통신 개발 경험을 통해 데이터를 주고받는 구조와 연결 방식을 이해
  • 솔루션 엔진을 리눅스 서버에 설치하고 로그 확인, 프로세스 점검, 설정 파일 관리 등을 수행하며 서버 기반 문제 분석 및 운영 역량 향상

아쉬운점

  • 기존 소스코드가 존재하지 않아 화면 및 기능 요건을 분석하는 데 프로젝트 초기에 많은 시간 소요
  • 향후에는 소스코드가 존재하지 않는 프로그램은 기존 시스템의 기능 명세서나 화면 정의서를 사전 정리하여 현업과 요구사항을 체계적으로 정리하는 과정을 먼저 진행하면 시간 단축 가능
